# USV 仿真平台开发指导文档

## 1. 文档目标

本文档用于指导一个面向研究与算法验证的 USV 仿真平台开发。该平台以 **Python + gymnasium** 为基础，以 **Fossen 3-DOF 平面运动模型** 为物理核心，用于支持后续多类算法的快速接入、验证和迭代，包括但不限于：

- 强化学习（RL）
- 扩散模型（Diffusion Model）
- 流匹配（Flow Matching）
- 人工势场（APF）
- 模型预测控制（MPC）
- 快速随机树（RRT / RRT*）
- belief control
- 控制障碍函数（CBF）
- 轨迹生成与跟踪类方法

该项目的设计重点不是“先把某一个算法跑通”，而是构建一个 **长期可扩展、算法中立、工程边界清晰** 的研究底座。

---

## 2. 核心设计目标

### 2.1 使用场景

场景由以下元素构成：

- 1 个进攻 USV（主体）
- 若干个防守 USV（数量在环境创建时指定）
- 固定障碍物
- 有限边界的海面空间
- 目标区域（在环境创建/重置时生成）

进攻 USV 的任务是：

- 感知场上态势
- 规避障碍
- 避免被防守 USV 捕获
- 最终到达目标区域

环境在每一步执行的基本闭环为：

1. 由环境根据当前真实状态生成进攻 USV 可观测信息
2. 进攻侧决策模块根据观测给出控制输入
3. 防守侧决策模块根据世界状态或其可用观测给出控制输入
4. 所有控制输入统一送入动力学模块
5. 完成系统状态步进
6. 计算奖励、终止条件、调试信息并返回

---

### 2.2 设计原则

本项目必须严格遵循以下原则：

#### 原则 A：先简单，再复杂
先建立可运行、可验证、可复现的最小闭环，再逐步增加复杂性。不要在项目初期同时引入：

- 复杂感知建模
- 多种船体模型
- 多种渲染系统
- 多智能体训练框架
- 离线数据管道
- 海流风浪扰动
- 大量实验管理工具

#### 原则 B：物理真值层稳定
真值仿真核心统一采用 Fossen 3-DOF 平面模型。后续不论接入 RL、APF、MPC、RRT、CBF 还是生成式轨迹模型，都不允许绕开或污染真值物理层。

#### 原则 C：任务层与算法层解耦
环境负责：

- 世界状态维护
- 动力学推进
- 观测生成
- 奖励计算
- 终止判定

算法负责：

- 依据观测或结构化状态输出动作、路径、轨迹或参考量
- 如有需要，在算法侧维护 belief、history、内部记忆状态

#### 原则 D：对算法中立
平台不能围绕某一种方法定制。环境核心不能因为后续要接入某种方法就写死特殊逻辑。正确做法是通过适配层支持多类算法。

#### 原则 E：可复现与可评估
所有随机过程必须受 seed 控制。每个 episode 都应可追溯其：

- seed
- 场景参数
- 初始位置
- 障碍布局
- 防守艇配置
- 终止原因
- 奖励分项

## 3. 项目边界与阶段性约束

### 3.1 第一阶段明确不做的内容

为避免过早复杂化，第一阶段不建议引入：

- 雷达/声呐原始感知仿真
- 部分可观测滤波器内置实现
- 高保真水动力耦合
- 6-DOF 动力学
- 复杂非凸障碍物碰撞几何
- 多智能体联合训练框架
- 分布式训练系统
- 复杂 GUI

### 3.2 第一阶段建议做到的内容

第一阶段目标是形成一个 **可用的最小研究平台**，至少包含：

- 单一真值动力学核心：Fossen 3-DOF
- gymnasium 风格环境接口
- 攻击艇 + 目标 + 边界
- 固定障碍物
- 至少一种防守艇策略
- 几何可见对象观测
- 基础奖励与终止逻辑
- 固定种子可复现
- 至少一个可跑通的 baseline

## 4. 总体架构建议

建议将项目拆分为六层。

### 4.1 数学与物理基础层
职责：

- Fossen 3-DOF 状态更新
- 坐标变换
- 角度归一化
- 数值积分
- 基础几何计算
- 碰撞检测
- 边界距离计算

特点：

- 不依赖 gymnasium
- 不依赖具体任务定义
- 不依赖某个算法框架
- 只负责正确、稳定地描述系统物理和几何关系

---

### 4.2 世界建模层
职责：

- 定义世界中的实体与状态
- 维护攻击艇、防守艇、障碍、目标、边界等对象
- 提供统一的世界状态结构，作为唯一事实来源

建议这里形成单一真值对象，例如 `WorldState`，后续：

- 观测由它生成
- 奖励由它计算
- 终止由它判定
- 日志由它导出

---

### 4.3 仿真编排层
职责：

- 管理 `reset()` / `step()` 生命周期
- 调度攻击侧动作和防守侧动作
- 调用动力学推进
- 执行事件检测
- 更新 episode 内部时钟和步数

这层相当于环境核心调度器，但不应该在这里堆叠具体奖励细节或算法逻辑。

---

### 4.4 任务定义层
职责：

- 观测设计与编码
- 奖励项设计与组合
- 终止与截断判定
- 成功/失败事件定义

这一层决定“做什么任务”，而不是“怎么推进物理世界”。

---

### 4.5 gymnasium 适配层
职责：

- 实现标准 `reset` / `step` / `render` / `close`
- 定义 `observation_space` 与 `action_space`
- 兼容训练框架、wrapper、vector env

建议 gym 环境本身尽量薄，仅做接口暴露和流程编排。

---

### 4.6 算法适配层
职责：

- 对接 RL policy
- 对接规划器
- 对接轨迹生成模块
- 对接轨迹跟踪控制器
- 对接安全过滤器（例如 CBF）

这一层非常关键，它决定平台能否长期支持不同类型算法而不改动核心环境。

---

## 5. 推荐目录结构

建议采用 `src` 布局，目录结构可参考：

```text
project_root/
├─ src/
│  └─ usv_sim/
│     ├─ envs/                 # gymnasium 环境入口
│     ├─ core/                 # WorldState、时钟、仿真编排、事件
│     ├─ dynamics/             # Fossen 3-DOF、积分器、动力学参数
│     ├─ entities/             # attacker / defender / obstacle / goal / boundary
│     ├─ geometry/             # 碰撞、距离、可见性、坐标变换
│     ├─ scenarios/            # 场景模板、采样器、curriculum
│     ├─ observation/          # 可见性筛选、观测编码
│     ├─ reward/               # reward term 与 reward composer
│     ├─ termination/          # terminated / truncated 判定
│     ├─ controllers/          # 低层控制器、跟踪器、APF等
│     ├─ planners/             # RRT / MPC / diffusion / flow adapters
│     ├─ adapters/             # planner/controller/action 的接口适配
│     ├─ logging/              # rollout recorder、metrics、episode summary
│     ├─ config/               # 配置加载、schema、校验
│     └─ utils/
├─ configs/
│  ├─ env/
│  ├─ dynamics/
│  ├─ scenario/
│  ├─ observation/
│  ├─ reward/
│  ├─ termination/
│  ├─ controller/
│  ├─ planner/
│  └─ experiment/
├─ tests/
│  ├─ unit/
│  ├─ integration/
│  ├─ regression/
│  └─ baselines/
├─ docs/
└─ devREADEME.md
```

说明：

- `controllers/` 用于“当前步输出控制命令”的方法
- `planners/` 用于“生成路径或轨迹参考”的方法
- `adapters/` 负责把不同方法输出映射到环境可执行动作
- `logging/` 用于为 RL、模仿学习、扩散模型和 flow matching 记录统一数据格式

## 6. 核心对象与抽象边界

以下为推荐保留的核心抽象，不要求初版全部实现完整继承体系，但建议从一开始就明确边界。

### 6.1 USV 状态

建议每个 USV 的状态至少包括：

- 平面位置：`x, y`
- 航向角：`psi`
- 船体坐标系速度：`u, v, r`

可选扩展：

- 上一时刻动作
- 加速度近似量
- 执行器状态
- 船体半径或几何尺寸
- agent id / team id

---

### 6.2 WorldState

建议维护统一世界状态结构，至少包含：

- 当前时间 `t`
- 当前 step index
- attacker 状态
- defenders 状态列表
- 障碍物列表
- 目标区域
- 边界定义
- 事件缓存
- 当前 episode 元数据

重要原则：**WorldState 是唯一事实来源**。

不要同时维护多份互相可能不一致的“简化状态”“观测状态”“训练状态”。

---

### 6.3 动力学模型接口

建议设计统一的动力学推进接口，其职责是：

- 接收当前状态与控制输入
- 按给定 `dt` 推进到下一状态
- 进行必要的数值稳定处理
- 返回更新后的状态

虽然当前确定使用 Fossen 3-DOF，但仍建议将参数和接口抽象出来，而不是把模型硬编码到环境主文件中。

---

### 6.4 动作表示

建议从一开始就区分不同层级的动作语义。

#### 底层动作
适合直接驱动真值动力学，例如：

- surge force / thrust
- yaw moment
- 或者简化的纵向推力 + 转向输入

#### 中层动作
适合控制器或规划器，例如：

- 目标速度
- 目标航向
- 目标角速度
- 目标 waypoint
- 短时参考轨迹

原因：

- RL 通常直接输出连续动作
- APF 常输出方向或期望速度/航向
- MPC 可能输出一段控制序列
- RRT 输出路径点
- 扩散模型/流匹配往往输出轨迹样本

因此环境层面建议默认只接受一种“执行级动作”，同时通过适配器接入其他方法。

---

### 6.5 观测接口

建议将观测过程拆成两个阶段：

1. **可见对象筛选**
2. **观测编码**

即：

- 先根据感知范围从 `WorldState` 中找出当前可见对象
- 再将可见对象编码为 gym 观测

这两步不要混在一起。这样未来从“直接几何观测”过渡到“带噪声、带遮挡、带延迟、部分可见”时，改动范围会更小。

---

### 6.6 奖励接口

奖励不建议写成单个大函数，而应拆为多个 reward term，再通过 composer 组合，例如：

- 到目标推进奖励
- 达到目标奖励
- 碰撞惩罚
- 被捕获惩罚
- 越界惩罚
- 时间惩罚
- 控制平滑惩罚
- 安全距离惩罚

要求：

- 每项可开关
- 每项权重可配置
- 每项可单独记录到 `info`

---

### 6.7 终止接口

建议统一输出：

- `terminated`
- `truncated`
- `reason`

务必区分：

- `terminated`：自然任务终止，例如成功、碰撞、被捕获、越界
- `truncated`：时间上限或外部截断

这对 RL 框架稳定训练非常重要。

---

## 7. 动力学模型设计建议

### 7.1 为什么选 Fossen 3-DOF

该项目应在简单模型和高保真模型之间取平衡。Fossen 体系下的 3-DOF 平面模型是一个合理折中，因为它：

- 比纯点质量或单车模型更贴近 USV 实际运动
- 比 6-DOF 更易于工程实现和算法验证
- 足以支持大多数二维规划、控制和 RL 研究任务
- 便于后续扩展扰动、参数差异和控制约束

### 7.2 初期建模建议

第一阶段建议：

- 保留 `x, y, psi, u, v, r`
- 使用统一控制输入形式
- 给出合理的速度/角速度/推力约束
- 使用固定积分步长
- 在工程上优先保证数值稳定和调试便利

### 7.3 时间步长建议

建议明确区分：

- `dt_env`：决策步长
- `dt_sim`：仿真积分步长

一个 `env.step()` 内部执行多个 `sim substeps`。这样做有几个好处：

- 数值更稳定
- 高动态状态变化更真实
- 控制器与规划器可以低频工作
- 后续 MPC / CBF 接入更自然

不要直接把算法决策频率当作物理积分频率。

## 8. 场景建模建议

### 8.1 场景基本元素

环境中的对象建议第一阶段采用如下简化表示：

- 海面边界：矩形区域
- 目标区域：圆形区域
- 障碍物：圆形为主
- 船体碰撞模型：圆近似

不建议一开始就使用复杂多边形障碍和精细船体碰撞几何。

### 8.2 场景模板优先于完全随机

初期建议先实现一组固定模板场景：

1. 空旷区域到达目标
2. 单障碍绕行
3. 双障碍窄通道
4. 单防守艇追击
5. 多防守艇拦截
6. 障碍与防守混合场景

模板跑通后，再引入参数随机化和 curriculum。

### 8.3 场景生成器职责

场景生成器应负责：

- 根据 seed 采样场景
- 生成起点、目标、障碍、防守艇初始状态
- 保证场景合法
- 输出可追踪的场景元数据

必须做的合法性校验包括：

- 初始对象不重叠
- 出生点与目标不越界
- 目标不被障碍覆盖
- 防守艇不会直接贴脸出生
- 场景不是明显无解

## 9. 观测系统设计建议

### 9.1 观测总流程

建议采用以下三段式流程：

1. `WorldState`：完整真实世界
2. `VisibleObjects`：根据感知范围筛选可见对象
3. `Observation`：编码成可供算法消费的输入

这会让平台未来更容易扩展到：

- 部分可观测
- 带历史记忆
- belief state
- 传感器噪声
- 遮挡建模

---

### 9.2 第一阶段可见性规则

根据用户当前需求，建议先采用最简单规则：

- 全向感知（360°）
- 有限感知半径
- 无障碍遮挡
- 无噪声
- 无观测延迟

后续可以逐步增加：

- 视场角限制
- 障碍遮挡
- 噪声与漏检
- 时间延迟

---

### 9.3 推荐观测内容

#### 自身状态
建议至少包含：

- 自身位置或 ego frame 下的参考位置表示
- 航向角 / 航向差
- `u, v, r`
- 到目标的距离
- 到目标的相对方位
- 到边界的距离信息
- 时间剩余比例或步数比例

#### 防守艇对象特征
对每个可见防守艇建议包含：

- 可见 mask
- 相对位置
- 相对速度
- 相对距离
- 相对方位
- 对方航向或航向差
- 几何尺寸

#### 障碍物对象特征
建议包含：

- 可见 mask
- 相对位置
- 相对距离
- 半径/尺寸
- 类型标记

---

### 9.4 观测编码建议

由于后续需要兼容 RL 与传统方法，建议：

- 内部保留结构化表示
- 对外默认输出 `gymnasium.spaces.Dict`
- 如有需要，再通过 wrapper 转成扁平向量

不要一开始就把所有对象信息压成一个大向量，否则：

- 可读性差
- 调试困难
- 不利于后续引入 attention / set encoder / graph 方法

---

### 9.5 对象数量可变问题

由于防守艇和障碍物数量可能变化，而 RL 通常偏好固定 shape，建议：

- 环境内部保留对象列表
- 对外设置 `max_defenders`、`max_obstacles`
- 不足部分通过 padding 补齐
- 使用 `mask` 指示有效对象

### 9.6 对象排序稳定性

这是一个非常重要的工程细节。

如果每一步都按距离排序，对象槽位会频繁交换，容易导致：

- 训练不稳定
- 时序模型难以跟踪对象身份
- 生成式模型序列对齐困难

建议优先采用：

- 固定 ID
- 固定槽位
- 或者稳定排序 + 明确对象 ID / mask

---

## 10. 奖励系统设计建议

### 10.1 奖励设计原则

- 优先少量、解释性强的奖励项
- 每个奖励项独立、可配置
- 避免过度 shaping
- 保证 reward 与任务目标一致
- 每步输出奖励分项日志

### 10.2 推荐基础奖励组成

#### 主奖励
- 到达目标：大正奖励
- 与障碍碰撞：大负奖励
- 被防守艇捕获：大负奖励
- 越界：负奖励

#### 辅助奖励
- 朝目标推进的增量奖励
- 安全距离保持奖励或惩罚
- 控制平滑惩罚
- 每步小的时间惩罚

### 10.3 不建议初期加入的复杂奖励

- 过多启发式 shaping
- 依赖未来轨迹的奖励
- 对某一种规划器定制的奖励
- 难以解释的多目标加权组合

### 10.4 奖励记录建议

每个 step 的 `info` 中建议至少保留：

- `reward_total`
- `reward_progress`
- `reward_goal`
- `reward_collision`
- `reward_capture`
- `reward_boundary`
- `reward_smoothness`

这样后续才能系统地分析训练或控制失败原因。

---

## 11. 终止与截断设计建议

### 11.1 terminated 条件

建议至少包含：

- 到达目标区域
- 与障碍碰撞
- 被防守艇捕获
- 明确越界
- 严重数值异常

### 11.2 truncated 条件

建议至少包含：

- 达到最大步数
- 达到最大仿真时间
- 外部实验控制要求提前截断

### 11.3 reason 字段

终止原因建议标准化，例如：

- `goal_reached`
- `captured`
- `obstacle_collision`
- `out_of_bounds`
- `time_limit`
- `numerical_failure`

后续评估、统计和数据筛选都会依赖这个字段。

## 12. gymnasium 环境设计建议

### 12.1 接口目标

环境实现应尽量遵循 gymnasium 标准语义，便于接入常见 RL 训练框架与 wrapper。

### 12.2 reset() 建议流程

建议 `reset()` 做如下工作：

1. 接收 `seed` 与 `options`
2. 初始化随机数生成器
3. 根据配置或场景模板生成场景
4. 初始化 attacker / defender / obstacle / goal / boundary
5. 做场景合法性检查
6. 初始化 episode 计数器、事件缓存和内部日志缓存
7. 构造初始 observation
8. 返回 `obs, info`

`info` 中建议包含：

- `seed`
- `scenario_id`
- 初始目标距离
- 初始最小安全距离
- 场景参数摘要

### 12.3 step() 建议流程

建议 `step()` 内部依次执行：

1. 接收攻击艇动作
2. 动作裁剪与合法性检查
3. 生成或查询防守侧动作
4. 在一个 env step 内执行若干 sim substeps
5. 更新所有运动体状态
6. 检查事件（碰撞、捕获、越界等）
7. 计算当前 observation
8. 计算 reward
9. 判定 `terminated` / `truncated`
10. 组装 `info`
11. 返回五元组

### 12.4 防守艇策略建议

防守侧控制不建议一开始就做成训练智能体。初期建议先用脚本策略，例如：

- 直接追击
- 目标方向拦截
- 巡逻 + 触发追击
- 简单势场避障追击

原因：

- 便于调试环境
- 便于构造可控难度
- 便于建立 baseline

## 13. 如何兼容 RL、传统规划控制与生成式方法

这一部分是整个项目的关键。

### 13.1 核心思想

统一物理环境，不统一算法实现。

环境只负责：

- 给出观测或结构化世界快照
- 接收执行级动作
- 推进物理系统
- 返回结果

算法可以有多种形式，但最终都应通过适配链落到环境动作上。

---

### 13.2 建议的三类算法接口

#### A. 直接动作型
适用于：

- RL policy
- 直接控制式 APF
- 简单反应式控制器

输入：当前观测
输出：当前步动作

#### B. 参考路径/轨迹型
适用于：

- RRT / RRT*
- MPC 上层路径生成
- 扩散模型轨迹生成
- 流匹配轨迹生成

输入：世界状态 / 场景信息
输出：路径点或短期轨迹参考

然后由 tracking controller 将参考量转成当前步控制命令。

#### C. 安全过滤型
适用于：

- CBF
- shielded policy
- 约束投影层

输入：nominal action / nominal reference
输出：安全修正后的动作

---

### 13.3 推荐统一适配链

建议统一抽象为：

`Planner or Policy -> Reference or Action -> Tracking Controller -> Safety Filter -> Env Action`

这条链可以覆盖绝大多数目标方法：

- RL：`Policy -> Env Action`
- APF：`Potential Field -> Reference Heading/Speed -> Controller -> Env Action`
- RRT：`Path -> Path Follower -> Env Action`
- MPC：`Control Sequence -> First Action -> Env Action`
- Diffusion / Flow Matching：`Trajectory Sample -> Tracking Controller -> Env Action`
- CBF：插入到任意 nominal controller 外层

这样做的优点是：

- 算法与环境解耦
- 方法之间可以组合
- 后续替换某个模块成本低

---

### 13.4 belief control 的支持思路

尽管第一阶段不做雷达感知，但 belief control 仍可兼容。建议：

- 环境只输出当前时刻的可见对象与 mask
- 如有需要，环境提供 history wrapper 或 frame stack
- belief state 由算法侧维护

不要把 belief estimator 写死在环境核心中。

---

### 13.5 对扩散模型和流匹配的前置考虑

这类方法通常依赖高质量轨迹数据，因此平台从一开始就应为数据记录留接口。建议后续支持记录：

- 世界状态序列
- 观测序列
- 动作序列
- 可见性 mask
- 轨迹参考
- 奖励序列
- 终止原因
- 场景配置

但注意：第一阶段只需设计好 logging / recording 接口，不必一开始就做完整数据集系统。

## 14. 配置系统建议

### 14.1 配置原则

配置系统需满足：

- 简洁
- 可校验
- 可组合
- 可复现
- 可版本管理

### 14.2 初期建议

建议第一阶段采用：

- YAML 配置文件
- Python dataclass / pydantic / 自定义 schema 做校验

不建议一开始就接入过重的配置管理框架。

### 14.3 推荐配置分组

建议至少拆分为：

- `env`：步长、substeps、episode 长度、渲染
- `dynamics`：Fossen 参数、输入约束、速度限制
- `scenario`：边界大小、障碍数量、目标范围、防守艇数量
- `observation`：感知半径、对象上限、编码方式
- `reward`：各奖励项权重
- `termination`：成功、碰撞、捕获、越界阈值
- `defender_policy`：防守艇控制参数
- `experiment`：seed、日志开关、训练/测试划分

### 14.4 配置校验重点

配置层必须校验：

- 各时间步长为正
- 动作上下界合法
- 对象数量上限合理
- 感知上限与观测维度匹配
- 障碍和目标生成范围合法
- 捕获半径、目标半径与边界尺度匹配

## 16. 测试策略建议

### 16.1 单元测试

重点覆盖：

- Fossen 3-DOF 数值推进正确性
- 坐标变换与角度归一化
- 圆形碰撞检测
- 边界判定
- 可见对象筛选
- reward term 输出正确性
- terminated / truncated 语义正确性
- 配置加载和校验

### 16.2 集成测试

重点覆盖：

- `reset()` / `step()` 是否符合 gymnasium 接口约定
- 固定 seed 下环境是否可复现
- 输出 observation 是否匹配 `observation_space`
- 动作是否匹配 `action_space`
- 一个 episode 能稳定执行完
- 防守侧脚本策略是否与环境兼容

### 16.3 回归测试

建议固定若干 baseline 场景与 baseline 策略，长期跟踪：

- 成功率
- 平均奖励
- 最小安全距离
- 平均到达步数
- 被捕获率

这样后续修改环境核心时能及时发现破坏性变更。

### 16.4 Baseline 很重要

项目早期至少建议实现三个 baseline：

- 直线/航向跟踪 baseline
- APF baseline
- 简单 defender pursuit baseline

baseline 的意义不在于性能，而在于确认：

- 环境是可控的
- reward 是合理的
- 任务难度是可调的
- 平台具备最基本的研究可用性

## 18. 工程上最容易踩的坑

### 18.1 坐标系混乱

USV 项目中最常见错误之一是：

- 世界坐标系与船体坐标系混用
- 航向角方向约定不一致
- 度与弧度混用

必须统一约定，并在整个项目中保持一致。

### 18.2 把算法逻辑写进环境

例如：

- 为某个 RL 模型特地改 observation
- 为某个规划器特地改 reward
- 为某个方法直接把中间量写进核心环境

这样会迅速破坏平台的扩展性。

### 18.3 观测泄漏

当前设定下，进攻艇只能看到“可见对象”。如果奖励、观测或调试路径里偷偷使用了全局不可见真值，就会导致：

- 训练结果虚高
- 算法评估不公平
- belief control 研究失真

### 18.4 对象槽位抖动

如果可见对象每一步都重新排序，神经网络和生成模型都会难以稳定建模。

### 18.5 terminated / truncated 混淆

这会影响 RL 训练和评估解释，必须严格区分。

### 18.6 奖励过度 shaping

初期 reward 越复杂，越容易让策略学偏。应先小而稳，再逐步增强。

### 18.7 场景生成缺少合法性检查

若场景生成器不做检查，训练会充满噪声和无效 episode。

### 18.8 决策频率与积分频率混淆

将控制周期直接用作物理积分周期，往往会造成数值不稳定和行为失真。

### 18.9 过早引入大而全框架

项目一开始最重要的是把“物理闭环 + 任务闭环”跑通，而不是追求全套平台化工具链。

---

## 19. 推荐的实际开发顺序

建议开发顺序如下：

1. 搭建 `WorldState + Dynamics + GymEnv` 最小骨架
2. 先做单进攻艇到目标任务
3. 加入边界与基础终止逻辑
4. 保证固定 seed 可复现
5. 加入障碍物和碰撞检测
6. 建立“可见对象 -> 观测编码”链路
7. 加入单个防守艇和脚本策略
8. 将奖励、终止、场景生成模块化
9. 增加 baseline：tracking / APF
10. 再扩展到 RRT、MPC、CBF、diffusion、flow matching

这个顺序的核心思想是：**先建立可靠世界，再接入复杂算法。**

## 21. 最终架构总结

最推荐的总体架构不是“为某种算法做一个环境”，而是：

> 以 Fossen 3-DOF 作为稳定的真值物理内核，
> 以 WorldState 作为唯一事实来源，
> 以 Scenario / Observation / Reward / Termination 作为任务插件层，
> 以 Planner / Controller / Safety Filter 作为算法适配层，
> 以 gymnasium 作为统一环境接口。

只有这样，这个平台才能在后续持续支持：

- RL
- 传统规划控制
- belief 类方法
- 扩散模型
- 流匹配
- 以及未来尚未确定的新算法形态

并且做到在不频繁重构环境核心的前提下持续演进。
